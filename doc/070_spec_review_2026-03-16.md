## 総評

仕様の方向性（「Live USB ではなくフル Linux を USB に移植」）は全体的に一貫しており、
GUI 導線・安全性・実装補助・試験観点まで文書化されている点は非常に良い。

一方で、**文書間での実装方式の不一致**（Python中心 vs shell中心）、
**運用上の重大リスクにつながる記述不足**（chroot コンテキスト、Secure Bootの成立条件）、
**受け入れ基準の現実性不足**（成功率100% など）があり、
v1 着手前に整合化が必要。

---

## 重要度: High（リリース阻害になり得る）

### H-01: 実装アーキテクチャが文書間で不一致
- `030` は PyQt6 + Python 4層アーキテクチャ前提。
- `050` は `src/lib/*.sh` 中心の shell モジュール構成前提。
- `020` は Python ファイル例（`disk.py`, `grub_setup.py`）を提示。

**懸念**
- 実装開始後に責務分割がぶれ、レビュー/テスト/パッケージングが破綻しやすい。

**提案**
- v1 の「正本」を 1 つ決める（例: GUI=Python、低レイヤー=shell を subprocess 呼び出し、など）。
- 他文書は「採用しない方式」を明記して誤読を防ぐ。

### H-02: Bootloader / initramfs 更新の実行文脈が曖昧
`020` の手順は `update-grub` の実行場所（ホスト/ターゲットchroot）が不明。
また machine-id 再生成例がターゲットルートへの反映になっていない。

**懸念**
- ホスト側設定を誤って更新する事故。
- コピー先 USB の起動失敗または ID 重複残存。

**提案**
- 「すべて target root に chroot して実行」を明記。
- 具体例を `chroot /mnt/usb ...` 形式で統一。
- machine-id は `040` の firstboot 方針に統一（作成時再生成ではなく初回起動時）。

### H-03: Secure Boot 対応要件が成立条件不足
`020` は Secure Boot 対応可能と記述するが、署名チェーン・鍵運用・配布要件（shim-signed 依存の前提）が不足。

**懸念**
- 「対応可能」の解釈が広すぎ、受け入れテスト不能。

**提案**
- v1 は「非対応」または「shim-signed 利用時のみ限定対応」に絞る。
- 対応時は前提条件（ディストリ、パッケージ、MOK 操作有無）を明記。

## 重要度: Medium（品質・保守性に影響）

### M-01: rsync 除外ポリシーの差分が複数文書で未統一
- `020` 基本除外リスト
- `030` 追加推奨（`/var/tmp`, APT）
- `040` は `-x`, `--numeric-ids`, `--delete` まで含む強化版

**懸念**
- 実装者ごとにコマンドが分岐し、再現性が下がる。

**提案**
- v1 正式コマンドを 1 本化し、GUI オプションで有効/無効が変わる部分だけ差分定義する。

### M-02: 用語・プロダクト名の表記ゆれ
`oYo` / `oyo`、`portable usb creator` / `portable system creator` が混在。

**懸念**
- パッケージ名、desktop ファイル名、ログキー、ドキュメント検索性に悪影響。

**提案**
- 正式表記を決め、命名規約を別紙化。

### M-03: firstboot の責務境界が一部文書で曖昧
`020` では作成時再生成、`040` では firstboot 再生成を推奨。

**提案**
- 一意に firstboot へ寄せる（実機差分・衝突回避の観点で妥当）。

---

## 重要度: Low（改善推奨）

### L-01: テスト受け入れ基準が現実的でない
`060` の「USB作成成功率100%」は、実機差分を考えると v1 ゲートとして過剰。

**提案**
- 目標値と必達値を分離（例: 必達95%以上、既知問題は除外条件を定義）。

### L-02: 非機能要件の定量化不足
ログ保持期間、メモリ上限、処理中断時のロールバック保証レベル等が未定義。

**提案**
- v1 の最低限 SLO を追加（例: 32GB USB で完了時間目安、失敗時の再試行可否）。

---

## 推奨アクション（優先順）

1. **実装方式の正本決定**（Python中心 / shell中心 / ハイブリッドの明文化）
2. **危険手順の実行文脈統一**（chroot・mount・umountの厳密手順化）
3. **Secure Boot 要件のスコープ縮小 or 明確化**
4. **rsync・firstboot の仕様一本化**
5. **テスト合格基準の現実化**

---

## レビュー結論

この doc セットは、製品コンセプト・UI 導線・安全配慮が充実しており、
v1 仕様の土台として十分有用。

ただし、**実装方式の不一致**と**ブート周辺の手順曖昧性**は重大であり、
ここを先に解消しないと、試作段階で設計の手戻りと検証失敗が高確率で発生する。
