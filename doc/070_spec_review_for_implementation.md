# 仕様レビュー（実装着手可否）

対象: `doc/` 配下の仕様書一式  
レビュー観点: 「実際にプログラムを作成するために必要な情報が揃っているか」

---

## 1. 結論（サマリ）

現状の仕様は**構想・設計意図・処理の大枠**は十分に揃っており、
PoC/プロトタイプ実装は開始できます。

一方で、**本実装（複数人開発・品質担保前提）に必要な“厳密仕様”が不足**しています。
特に以下が未確定です。

- CLI/GUI の入力仕様（引数、バリデーション、既定値）の厳密定義
- 失敗時の復旧ポリシーの実装レベル詳細（どこまで自動で戻すか）
- firstboot / 最適化の適用範囲と ON/OFF 条件
- ログフォーマット・エラーコード・終了コードの標準化
- テストの自動化条件（実行コマンド、合否判定、再現性）

=> **評価: 65〜75% 程度の実装準備度**（手を動かせるが、着地品質が人依存になりやすい）。

## 2. 揃っている情報（実装に有効）

### 2.1 プロダクトの目的・スコープ
- 「Live USB ではなく通常 Linux を USB へ移植」という方針が明確。
- v1 の対応/非対応（LVM/mdraid/dm-crypt 非対応、Secure Boot は v1.1 以降）が定義済み。

### 2.2 技術方針
- Python 実装（GUI: PyQt6、実行ロジックも Python）とディレクトリ正本が整理されている。
- 主要処理（partition, rsync, grub, initramfs, firstboot）の実施ステップが明記されている。

### 2.3 安全方針
- root disk 誤選択防止、容量チェック、非対応構成の拒否が仕様として存在。
- 失敗時ポリシーや cleanup の考え方が示されている。

### 2.4 テスト観点
- 正常系/異常系/性能/回帰の観点が網羅されている。
- 受け入れ閾値（120分、メモリ 512MB、ログ 50MB 目安）がある。

## 3. 不足している情報（実装前に追加すべき項目）

以下は**実装のブレ・手戻りに直結**するため、優先して補完を推奨します。

### A. インターフェース仕様不足（最優先）

1. CLI 引数仕様の不足
- 必須/任意引数、排他条件、デフォルト値、終了コードが未定義。
- 例: `--device` 以外（`--mode`, `--include-wifi`, `--include-ssh-keys`, `--no-optimize`, `--dry-run`）の扱いが未定。

2. GUI 項目の値仕様不足
- 「推奨設定」「詳細設定」はあるが、各項目の許容値・保存形式・既定値が厳密化されていない。

3. ログ/エラーの外部契約不足
- エラーコード体系（例: E101=デバイス検証失敗）やユーザー向けメッセージID未定義。

### B. 処理仕様の詳細不足（高優先）

4. パーティション仕様の境界条件
- セクタ境界・アラインメント・最小容量計算の式が未定義。
- 既存パーティションテーブル破壊前の最終確認条件が曖昧。

5. rsync 仕様の差分
- 文書間で除外候補・追加候補の表現があり、v1固定の最終リストが明文化されていない。
- `--delete` の適用条件（再実行時のみ or 常時）を確定させる必要あり。

6. firstboot の実行責務
- どのファイルを build 時に配置し、どこまで firstboot で実行するかは方針があるが、
  失敗時再試行条件・完了マーカー更新順序が未定義。

### C. 運用/品質保証仕様不足（高優先）

7. テスト自動化不足
- 手動確認中心で、CI で回せる単体/結合テストの責務分担が未定。

8. サポート環境の粒度不足
- Debian/Ubuntu の具体バージョン、GRUB パッケージ差異、BIOS/UEFI 実機条件が粗い。

9. セキュリティ・機密情報取り扱い
- SSH鍵 OFF 既定はあるが、機密対象の完全一覧とマスキング規則が不足。

- [ ] GUI オプション仕様表（画面項目ID、保存キー、既定値、制約）

- [ ] パーティション作成ルール（最小容量式、アラインメント、失敗時扱い）

- [ ] エラーコード一覧（ユーザー表示文言と内部ログ文言）
- [ ] 自動テスト範囲（unit/integration/e2e）と最低本数
