# doc仕様レビュー（実装着手可否）

## 結論
- **実装の方向性は十分に定義されており、プロトタイプ開発は開始可能**。
- ただし、**本実装（安全に配布できる v1）に必要な決定事項が未確定**のため、現状のままでは実装中に仕様解釈が分岐しやすい。
- 特に不足しているのは、
  1) **ソース構成の正本統一**、
  2) **異常系/ロールバック仕様**、
  3) **対象外ディストリや特殊構成（LVM等）の扱い**、
  4) **受け入れ判定の定量化**。

---

## 良い点（実装に必要な情報が揃っている部分）

1. **プロダクトの目的と非採用方針が明確**
   - Live USB系（overlay/persistence）ではなく、通常 Linux として動かす方針が明示されている。

2. **主要処理フローが具体的**
   - パーティション（EFI 512MB + root ext4）、rsync、fstab更新、GRUB、initramfs まで流れが記載されている。

3. **安全要件の方向性が強い**
   - root disk 除外、USB候補判定、チェックフローが複数文書で強調されている。

4. **GUI導線が詳細**
   - ウィザード画面の目的、表示項目、確認チェックが定義され、実装に落とし込みやすい。

5. **テスト観点の網羅性がある**
   - デバイス検出〜起動確認〜firstboot まで、最低限の観点が揃っている。

## 不足/曖昧な点（実装前に確定したい項目）

### A. 文書間で「正本」が分岐している
- ディレクトリ構成が文書ごとに異なる。
  - `src/lib` 前提の記述
  - `src/system` / `src/utils` 前提の記述
  - `src/services` / `src/infra` 前提の記述
- **影響**: 実装担当ごとにファイル配置がずれ、レビュー・CI・パッケージングで衝突する。
- **必要対応**: 「v1正本構成」を1つに固定し、他文書から参照する形に統一。

### B. 失敗時の復旧・ロールバック仕様が未確定
- `rsync失敗/GRUB失敗/mount失敗` は挙がっているが、
  - 途中失敗時に target をどう扱うか（再試行可否、未完成印の付与、再実行時動作）
  - どこまで自動アンマウント・クリーンアップするか
  が未定義。
- **影響**: USBが半端状態で残り、再実行時に破壊的な挙動差が出る。
- **必要対応**: ステップごとの失敗ポリシー（abort/continue/cleanup）を表形式で定義。

### C. 対応範囲の境界条件が弱い
- LVM / mdraid / dm-crypt の言及はあるが、**v1で対応するのか検出して拒否するのか**が不明。
- `tran==usb` を優先としつつ `rm` 補助利用の方針があるが、USBブリッジ例外時の最終判断ルールが未確定。
- **影響**: 誤検出または「動く環境だけ動く」実装になりやすい。
- **必要対応**: サポートマトリクス（対応/非対応/警告許可）をOS構成別に作成。

### D. firstboot設計の競合/責務分離が曖昧
- machine-id / SSH鍵再生成は定義済みだが、
  - Wi-Fi引き継ぎ、ユーザーキャッシュRAM化、ブラウザキャッシュ切替などを
    build時に行うか firstbootで行うかの境界が曖昧。
- **影響**: 起動前準備と初回起動処理が重複し、バグ温床になる。
- **必要対応**: 「ビルド時に確定する処理」「firstboot限定処理」の一覧を固定。

### E. 性能・完了判定の定量基準が不足
- テスト仕様に成功率はあるが、
  - コピー所要時間の閾値
  - RAM化オプション時のメモリ使用上限
  - ログサイズや保存期間
  など運用判断基準が未定義。
- **影響**: 完了条件を満たしても実運用で遅い/不安定な可能性が残る。
- **必要対応**: 最低性能基準（例: 32GB USBでシステム20GB時の目安時間）を追加。

### F. セキュリティ/プライバシー同意の仕様が不足
- SSH鍵やWi-Fi設定の引き継ぎオプションはあるが、
  - 具体的に何をコピーするか（パス一覧）
  - マスク表示・警告文
  - 監査ログ項目
  が不足。
- **影響**: 意図しない秘密情報の持ち出しや説明責任不足。
- **必要対応**: 「機密データコピー仕様」章を追加。
